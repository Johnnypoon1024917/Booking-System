How permissions work

Every meaningful action is gated by two layers:

  1. Role gate — coarse, controls what menus appear (e.g. only admins see /admin/*).
  2. Permission gate — fine-grained, checks one of the 27 permission keys before the API will perform an action.

Role-to-permission mapping is stored per tenant in the role_permissions table and is editable through Settings → Permissions. Default mappings are seeded by migration 007_integrations_permissions.up.sql.

Tenants can rebalance the matrix without code changes. A "System Admin" in one tenant might lack tenant.manage in another tenant if the operator chose to lock that down.

The 5 built-in roles

System Admin

Full system access

Holds every permission by default. Can edit tenant branding, manage users, configure approval chains, view audit log. Use sparingly in production.

RoleSystemAdmin

Security Admin

Manages permissions & audit

Read-only across bookings, can decide approvals (VIP/Top Management), edits the permission matrix and views the audit log. Cannot create resources or tenants.

RoleSecurityAdmin

Room Admin

Regional room manager

Owns the rooms in their assigned region_access regions. Can edit room metadata, split composite rooms, decide approvals for their region, manage holidays.

RoleRoomAdmin

Secretary

Top Management gatekeeper (SDO-grade)

Only role that can see resources marked asset_type = 'Top Management' (DGFS bookings). Books on behalf, approves Tier-1 of the VIP chain. Must hold grade='SDO'.

RoleSecretary

General User

Standard staff

Self-service booking, cancellation, and updates of their own bookings. Cannot see restricted resources, cannot decide approvals, no admin access.

RoleGeneralUser

Test accounts & how to log in

The dev/demo setup ships with a directory simulator (active when LDAP_URL is unset). It hard-codes three accounts so you can sign in immediately after docker compose up -d. Two more roles (Security Admin, Room Admin) don't have ready-made accounts — you'll create them through the admin UI in Step 0 below.

adminadmin123System AdminGrade SDO · access to all 3 regions
secretarypassSecretaryGrade SDO (required for Top Management visibility)
officerpassGeneral UserRegion access: Hong Kong only
Production reminder: these simulator credentials disappear as soon as LDAP_URL is configured. In production, authentication is delegated to your AD/LDAP server.

Step 0 — Create Security-Admin and Room-Admin test users

  1. Log in as admin / admin123 at http://localhost:8080/app/login.
  2. Click the Settings icon in the sidebar → click Users in the admin sub-nav.
  3. Find the existing rows (or use the simulator to log them in once so they're upserted), then click the pencil icon. Set role:
    • secretary → already Secretary, leave it
    • Pick any other user and reassign to Security Admin for testing
    • Pick another user and reassign to Room Admin, set Region Access = ["Hong Kong"]
  4. Or add two new lines to the simulator at src/infrastructure/ad/ldap_service.go for permanent test users: 
    if username == "sec" && password == "pass" {
    return &user.User{ID: "AD-3001", Username: username,
        Role: user.RoleSecurityAdmin, IsActive: true,
        DN: "CN=Sec Officer,OU=Security,DC=fsd,DC=gov,DC=hk"}, nil
}
if username == "roomadmin" && password == "pass" {
    return &user.User{ID: "AD-3002", Username: username,
        Role: user.RoleRoomAdmin, IsActive: true,
        RegionAccess: []string{"Hong Kong"},
        DN: "CN=Room Admin,OU=Facilities,DC=fsd,DC=gov,DC=hk"}, nil
}
    Then docker compose up -d --build mrbs_api to reload.

Test plan — System Admin

System Admin

admin / admin123

Expected sidebar

All 8 items visible: Dashboard · Schedule · New Booking · All Bookings · Report · Approvals · Broadcast · Settings.

Test cases

  1. Create a room — Settings → Rooms → + Create. Fill Name "Test Room A", Capacity 10, Region "Hong Kong", Location "Test Building 1/F", click Save.
    A new row appears in the Rooms list with status "Active". Permission gate: resource.create.
  2. Edit room equipment — Click the pencil icon on "Test Room A", set Equipment = ["Projector","Whiteboard"], save.
    Equipment chips render. Permission gate: resource.update.
  3. Split a composite room — Create a Composite parent room (Capacity 80, set Composite mode = Parent), then split into 2 child sub-rooms.
    Parent + 2 children render in Rooms list. Permission gate: resource.split.
  4. Cancel another user's booking — Log in as officer in another browser, create a booking. Back in admin browser, Settings → All Bookings → click the cancel icon on that booking.
    Status changes to Cancelled. Permission gate: booking.cancel_others.
  5. Edit tenant branding — Settings → Tenant Studio → change brand primary colour to #dc2626, save.
    Sidebar accent shifts to red after reload. Permission gate: customization.manage.
  6. Open the audit log — open the URL /api/v1/admin/audit (or via API).
    Returns 200 with audit events from the actions above. Permission gate: audit.view.
  7. Import HK holidays — Settings → Tenant Studio → click Sync Holidays.
    Holiday list under Settings → Holidays populates with the next year of gov.hk dates. Permission gate: holiday.import.
  8. Manage the permission matrix — Settings → Permissions → toggle a checkbox (e.g. give General User approval.decide), save.
    Save succeeds. Permission gate: permission.manage. Revert before continuing.

Customising the permission matrix

The matrix is editable per tenant under Settings → Permissions. Changes are persisted to role_permissions immediately and take effect on the next request (no restart needed).

Workflow

  1. Log in as System Admin or Security Admin (both hold permission.manage).
  2. Click Settings → Permissions.
  3. The matrix renders 9 groups × 5 roles. Tick or untick checkboxes.
  4. Click Save. The audit log records who changed which key for which role.
  5. Refresh the affected user's session (re-login) so their JWT picks up the new role granting.
Tip: when locking down a permission, grep the codebase first for the key to find every site it gates — sometimes a button is dynamically shown based on a role match, not the permission match. UI hiding is cosmetic; the API gate is the authority.

API verification (curl)

The cleanest way to prove a permission gate works is to bypass the UI and hit the API directly. Each test below should return 200 for granted users and 403 for non-granted.

1. Get a token

curl -X POST http://localhost:8080/api/v1/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}'

# response: { "token": "eyJhbGc..." }
export TOK=eyJhbGc...

2. Hit a gated endpoint

# Should be 200 for System Admin, 403 for General User
curl -i http://localhost:8080/api/v1/admin/audit \
  -H "Authorization: Bearer $TOK"

# Should be 200 for anyone who holds booking.create
curl -i http://localhost:8080/api/v1/bookings/search?location=Hong+Kong \
  -H "Authorization: Bearer $TOK"

# Should be 200 only for permission.manage holders
curl -i http://localhost:8080/api/v1/admin/permissions \
  -H "Authorization: Bearer $TOK"

3. Inspect the JWT to see your role

echo $TOK | cut -d. -f2 | base64 -d 2>/dev/null
# { "sub":"...","tenant_id":"...","role":"System Admin","grade":"SDO",
#   "regions":["Hong Kong","Kowloon","New Territories"], "exp":..., "iat":... }

4. Dump the current matrix

curl -s http://localhost:8080/api/v1/admin/permissions \
  -H "Authorization: Bearer $TOK" | jq .

5. Cross-check Quickly

Run this loop to verify every role × endpoint combo (paste into a terminal):

for U in admin:admin123 officer:pass secretary:pass; do
  USR="${U%:*}"; PWD="${U#*:}"
  TOK=$(curl -s -X POST http://localhost:8080/api/v1/login \
        -H "Content-Type: application/json" \
        -d "{\"username\":\"$USR\",\"password\":\"$PWD\"}" | jq -r .token)
  for EP in admin/audit admin/permissions bookings/search?location=Hong+Kong; do
    CODE=$(curl -s -o /dev/null -w "%{http_code}" \
            http://localhost:8080/api/v1/$EP \
            -H "Authorization: Bearer $TOK")
    echo "$USR  $EP  $CODE"
  done
done

admin     admin/permissions               200
admin     bookings/search                 200
officer   admin/audit                     403
officer   admin/permissions               403
officer   bookings/search                 200
